Distributed Computing for Machine Vision
About

The goal of streamproc is to build pipelined processes of distributed nodes for the purpose of performing complex machine vision tasks such as object tracking, person identification, object counting, etc. This documentation describes the C++ API of the library.

Foundations

Before diving into the functionality of the streamproc API, it is essential to understand the basic concepts which form the foundation of the API design.

Nodes

Achieving a goal such as threat detection requires a single video frame to undergo many stages of transformation - inference to image embedding to re-identification, etc. Performing this kind of transformation in a single function or class would not be maintainable. It would also be wasteful of time to perform only one stage at a time Running each stage concurrently would improve the overall performance of the system by increasing throughput. Moreover, even if each of these stages are isolated into their own thread, these tasks require a lot of resources. In situations where performance is necessary, such as real-time systems, it's likely not possible to achieve a desired benchmark on a single machine.

In order to address these challenges streamproc uses the concept of a node. A node is a process which executes an atomic task which takes part in a larger more complex task.

Stream

The concept of a node breaks down a complicated machine vision process into several simpler processes which work together. However, if all of these simple processes execute on the same machine, resources get exhausted. In order to address this challenge, nodes are distributed across a network of machines. A stream allows distributed nodes to communicate with each other. Each node specifies a set of streams where it can receive inputs from other nodes through an incoming stream and send outputs to other nodes through an outgoing stream.

The stream specifications of a node create a new layer of abstraction in the API design, where each node has a certain responsibility depending on the kinds of incoming and outgoing streams it owns.

Source

A source is a type of node which generates data and sends it to other nodes. It specifies a set of outgoing streams where this generated data is outputted.

Examples of sources:

  • File
  • Livestream
  • Camera

Filter

A filter is a type of node which sends filtered data to other nodes. It specifies a set of incoming streams where it receives inputs, as well as a set of outgoing streams where its sends filtered outputs.

Examples of filters:

  • Inference
  • Image Embedding
  • Semantic Segmentation

Sink

A sink is a type of node which consumes data that it receives from other nodes. It specifies a set of incoming streams where consumable data is inputted.

Examples of sinks:

  • File
  • Livestream
  • Alert System

Tutorial

Tech Stack

The streamproc API integrates with influxdb - a time-series database. This database aggregates time series information about videos. In addition to influxdb, the streamproc API also integrates with grafana - this is graphing ui for influxdb. This tech stack can be run using docker-compose.

Configuration

Nodes are configured using JSON files. The streamproc binary sp searches for json files in the current working directory and attempts to automatically build streamproc nodes from the configurations. Each streamproc node is equipped with a JSON schema which follows the JSON schema standard. The schemas are located in streamproc/json/schema.

Environment

Clone the repo, build, and bring up the entire stack (backend, db, frontend).

docker-compose for aarch64 is available here (binary and Dockerfile).

git clone https://<username>:<authkey>@github.com/arcturusnetworks/streamproc
cd streamproc
git submodule update --init
# Build depedencies (speeds up subsequent builds)
docker-compose -f docker-compose.init.yml build
# Build and run the entire stack (streamproc, influxdb, grafana, watchman)
docker-compose up -d --build

You can also use the following convenience script to automate the above two commands:

./tools/docker/build.sh

Watchman analytics interface will be available at http://<ip_address>.

The default configuration may also create an mjpegstream at <ip_address>:7000. The stream will show a video with annotations of object detections, people count, and node fps.

View the grafana dashboard at <ip_address>:3030 to see different metrics. The default dashboards display node fps, person count, and detection confidence.

Check the status of the running containers, and view logs by

# check status
docker-compose ps
# View logs in real-time
docker-compose logs -f

Rebuilding

After making changes to the source code you need to rebuild.

docker-compose up --build

NOTE: If any of the dependencies are updated (example: deepvision) their corresponding images also need to be updated before rebuilding. To make your life easy just re-run the following script which will ensure everything is built and kept up to date.

./tools/docker/build.sh

Documentation

To generate documentation, issue the following commands inside the streamproc container:

cd /usr/local/share/streamproc
sudo doxygen Doxyfile

The docs will be generated in the /usr/local/share/streamproc/html folder.
